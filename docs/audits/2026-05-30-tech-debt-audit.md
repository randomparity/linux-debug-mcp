# kdive Technical-Debt Audit — 2026-05-30

Consolidated from 119 raw findings across 13 subsystem auditors. No code was changed. (The `server.py` lines 1–3300 auditor was re-run after an initial structured-output failure; its findings — TD-99 through TD-110 — are merged in below.)

## Summary

Five themes dominate the debt surface:

1. **`server.py` is a ~9.9k-line god-file.** All three scoped thirds (1–3300 wiring/build/boot/probe, 3300–6600 test/introspect/debug, 6600–9922 debug.*) independently report the same shape: oversized handlers (`target_boot_handler` ~370 lines, `_debug_operation_response` 155+ lines), boilerplate handler families (5 execution handlers, 4 read handlers) that exist only to dispatch to a generic method, and copy-pasted scaffolding — three identical admission-rollback `except Exception` blocks (TD-08), three near-identical manifest-lock retry loops whose own docstrings admit they are clones (TD-99), and three drifted SSH-result-parsing ladders (TD-100). The size is the systemic issue; the boilerplate and copy-paste enforcement points (legacy-session fencing duplicated, `_BREAKPOINT_MUTATORS` manually kept in sync with `DEBUG_METHOD_OPERATIONS`) are symptoms. Much of this is sanctioned by ADRs 0021/0023/0024 but still raises drift risk. The first third also carries an inert `force_rebuild` flag (TD-109) and dense unparameterized probe tunables — the `+10` SSH grace and `cap=256` redaction limit repeated 6× and ~15× respectively (TD-104/105).

2. **The same logic is implemented twice in coordination.** Backend reaping (`transaction.py:95-99` vs `388-390`) appears in both the production teardown path (`_SessionSubscriber.invalidate`) and the open-failure unwind (`TransportTransaction._rollback`), each with an identical `contextlib.suppress(Exception) + getattr(transport, '_proxy', ...)` private-attribute reach. Two auditors flagged this from both ends; it is one finding with two call sites that should delegate to a single `reap_backend()` hook. The session-id-fenced `delete_record` TOCTOU is likewise reported twice (registry and transaction) as the same accepted-but-unguarded race.

3. **Redaction is relied upon but not enforced where non-string secrets flow.** `safety/redaction.py:46` requires the value to be a `str` before redacting under a secret-named key, so `{"password": 12345}` and secret-bearing lists/dicts leak verbatim into logs and artifacts. This is the single highest-severity finding because redaction is a mandatory trust-boundary invariant on every test/debug path. A secondary `server.py:3527` note (Redactor seeded only with `ssh_key_ref`, reused for preflight stderr) compounds it.

4. **Untrusted `int()`/`encode()` conversions raise unhandled exceptions at the transport boundary.** `qemu_gdbstub.py:56`, `serial_local.py:486-487` convert `transport_ref.opts`/`target_ref` ports and baud with bare `int()`, and `rsp_probe.py:10` does `encode('ascii')` on RSP payloads — all crash with `ValueError`/`UnicodeEncodeError` instead of a wrapped `CONFIGURATION_ERROR`. Parallel ELF-parsing gaps (`symbols/build_id.py:82`, `prereqs/drgn_probe.py:312`) lack bounds checks on `e_phnum`.

5. **Unsanitized environment variables flow into shell/filesystem in `build-rootfs.sh`.** `SSH_USER` is interpolated unquoted into a `virt-builder --run-command` guest shell (`:114`), and `ROOTFS_PATH` reaches `mkdir`/`chmod` without canonicalization or containment (`:25`). These cross a real trust boundary (env vars) into command execution and permission changes.

**Pervasive low-severity pattern:** undocumented magic timeouts and byte caps are scattered across nearly every module (providers, prereqs, transport, seams, coordination). Individually trivial; collectively the largest count. They are grouped into a single quick-wins sweep rather than padded into many rows.

**Invariant sanity-check (TASK 3):** Several findings touched deliberate project invariants and were downgraded/annotated rather than reported as defects:
- `server.py:6290-6291` (read_memory missing value bounds) — kept at **Medium** but annotated: the 4096 cap is enforced by the gdb/MI backend per the documented invariant; only the missing `address >= 0`/`byte_count > 0` *input* check is real debt.
- `gdb_mi.py:680` (hardcoded "1 and 4096" message) — kept **Low**; the cap itself is correct, only the literal-vs-constant drift is debt.
- `providers/base.py:6-33` (the unused capability-factory helper) — **not** treated as dead code to delete; the future-stub surface is intentional per the architecture. Downgraded to a documentation/confirmation note.
- `crash_parsers.py:114-131` bare-except passthrough — explicitly ADR 0026 decision 3; kept **Low**, remediation reduced to "add DEBUG logging" only.

No findings were dropped outright. No in-scope category yielded nothing — every category (1 explicit markers, 2 dead code, 3 parallel paths, 5 structural, 6 test/docs, 7 untrusted input, 8 path safety, 9 magic numbers, 10 unicode) produced at least one final finding.
