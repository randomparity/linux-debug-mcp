**Status:** Accepted (2026-05-29) Â· **Issue:** #82 (Phase D of #13; epic #9) Â· **ADR:** extends [0021](0021-gdb-mi-phase-c-session-registry-and-execution-state.md) (the guaranteed-resume teardown and the bounded interactive verbs) Â· **Affects:** `providers/gdb_mi.py` (`set remotetimeout` at attach, a bounded connect retry/backoff, and a distinguishable transport-stall signal on the timeout path), the per-op debug handlers in `server.py` (a stall reaps the session and runs the guaranteed-resume teardown rather than returning a retryable contained error).

## Context

Phase C bounded every interactive verb so a free-running kernel can never hang the tool call: each MI write carries `_MI_COMMAND_TIMEOUT_SEC` and each `*stopped` wait is deadline-bounded. But two robustness gaps remain for transports rougher than the clean QEMU gdbstub:

1. **gdb's own RSP read timeout is unset.** gdb defaults `remotetimeout` to 2s and, on some builds, retries indefinitely on a packet the stub never answers. Over a demuxed serial line the stub can be slow or momentarily silent; without an explicit, generous-but-finite `remotetimeout`, a single sluggish packet either spuriously drops the connection or stalls inside gdb below the MI timeout's visibility.
2. **A transport stall is currently indistinguishable from a benign command error.** When pygdbmi's `write` times out, the engine raises `GdbMiError(DEBUG_ATTACH_FAILURE)` â€” the same shape as a bad-symbol `^error`. The per-op handler treats a `GdbMiError` as a *contained* failure: it returns the error but **keeps the live session** so the agent can retry or `end_session`. That is correct for a bad symbol; it is wrong for a dead RSP link, where the target may be left HALTED behind a session that will never make progress. Phase D's acceptance is "an induced transport stall is **reported (not hung) and the target is resumed**" â€” which requires the stall to drive the guaranteed-resume teardown, not sit as a retryable error.

## Decision

### 1. Set `remotetimeout` explicitly at attach, before the RSP connect.

`attach()` issues `-gdb-set remotetimeout <RSP_REMOTE_TIMEOUT_SEC>` (a fixed, generous bound) immediately before `-target-select remote`, so every RSP packet gdb waits on is finite and the wait is owned by gdb (which can report a clean disconnect) rather than blocking opaquely under the MI write timeout. The value is a single module constant, set once.

### 2. Bound the RSP connect with a small retry/backoff for *transient* connect failures only.

`-target-select remote` is retried up to a fixed small count with a fixed backoff. **Classification is by idempotency, not by gdb error-text matching.** A failed `-target-select remote` surfaces over MI as a `^error` *message string* (e.g. `"â€¦: Connection refused."`), whose exact text is gdb-build/locale-dependent and brittle to pattern-match; so the engine does **not** try to distinguish "transient" from "permanent" by substring. Instead it relies on the connect being **idempotent** â€” no target state is mutated until `^connected` â€” and retries **any** `^connect error` a fixed small number of times, then fails. Over-retrying a genuinely permanent failure (bad endpoint, stub never coming up) costs only `count Ă— backoff` before the same clean failure is reported, which is bounded and harmless; under-retrying a transient race is the real cost the retry removes. Once `^connected` is observed, the connect is not retried again. The backoff sleeps are injectable (a `sleep` seam defaulting to `time.sleep`) so unit tests assert the retry count and ordering without wall-clock delay. The retry covers only the pre-`^connected` connect; it never re-issues a mutating or interactive verb (re-running `-exec-continue` after a partial stall would double-resume).

### 3. A timeout on an established session is a distinguishable transport stall that triggers guaranteed-resume teardown.

The engine's timeout path raises a `GdbMiError` carrying `details={"code": "transport_stall", ...}` and category `INFRASTRUCTURE_FAILURE` (not `DEBUG_ATTACH_FAILURE`), so the handler can tell "the link stalled" from "gdb rejected a command."

**"Established session" is a structural property, not a runtime flag.** `PygdbmiController.write` (`gdb_mi.py:189-197`) raises one uniform `GdbMiError` on *every* MI write timeout, including the connect-phase writes inside `attach()` (`-file-exec-and-symbols`/`-target-select`, `gdb_mi.py:262-263`) that run before `^connected`. To avoid mis-tagging an attach-phase timeout as a mid-session stall (which would tear down a session that never attached), `attach()` **wraps its own connect-phase writes and re-tags their timeouts as `DEBUG_ATTACH_FAILURE`** (feeding decision 2's connect retry / a clean attach failure), so the `code="transport_stall"` tag is reached **only through the per-op path** â€” which by construction runs after `register()`, i.e. post-`^connected`. "Established" therefore means "reached via a per-op handler," a structural guarantee, not an unstated boolean.

**Two stall shapes, both detected.** A stall can present two ways, and the acceptance's "induced stall is reported (not hung)" covers both:
1. **Write-path stall** â€” an MI command write times out (the link is dead enough that gdb cannot send the packet). `PygdbmiController.write` raises, tagged `transport_stall`, â†’ teardown.
2. **Silence-path stall** â€” the more realistic serial case: `-exec-continue` returns `^running`, then the link goes silent and `controller.read()` returns `[]` every slice (`gdb_mi.py:199-204` suppresses the read timeout). This is **indistinguishable from a healthy free-running kernel with no breakpoint** *until the interrupt fallback runs*: the timeout fallback issues `-exec-interrupt`, and a kernel that is actually reachable **cannot ignore a delivered SIGINT** â€” so "interrupt issued, write accepted, but no `*stopped` within `_INTERRUPT_STOP_TIMEOUT_SEC`" is itself the stall signal. `resume()`'s post-timeout interrupt therefore distinguishes: a `*stopped (SIGINT)` arrives â†’ benign `timed_out=true` (free-running, session retained, Phase-C behaviour); **no** stop arrives after a successful interrupt write â†’ raise `transport_stall` â†’ teardown. A benign timeout (`timed_out=true`, session kept) and a silence stall (`transport_stall`, torn down) are thereby separated by the interrupt's observability, not conflated.

**The stall runs the FULL transport teardown, not the per-op raw-fault path.** The existing per-op raw-fault branch (`server.py:4844-4855`) only reaps the attachment and calls `force_resume` â€” `force_resume` kills the gdb engine (so QEMU resumes the *guest*), but it does **not** write the durable Layer-4 record back to EXECUTING and does **not** release the `StopCapableGuard`. If the stall path did only that, the guest would resume but the durable record would stay HALTED and the guard held â€” `target.run_tests` gated indefinitely and the transport session leaked. So the stall path must run the **same full teardown `debug.start_session`'s probe-failure path uses** (`server.py:4111-4130`): `force_resume` (resume the guest) **then** `_resume_debug_transport` (durable EXECUTING write) **then** `_teardown_debug_transport` (release the guard + close the transport). "The target is resumed" means resumed at *both* the guest and the Layer-4 record, with the guard released.

**Detection and teardown happen INSIDE `debug_lock`.** The Phase-C `except GdbMiError` handler sits *outside* the `with store.debug_lock(run_id)` block (`server.py:4884`), so reaping there would race a concurrent same-run op that could `gdb_mi_sessions.require()` the still-registered stalled attachment after the lock released. The stall is therefore caught and torn down **within the inner `try` that already holds the lock** (alongside the raw-fault branch at `server.py:4842-4855`), reaping before the lock is released; only the structured failure response is built after. The handler returns an `INFRASTRUCTURE_FAILURE` report whose `suggested_next_actions` route the agent to `debug.start_session` (re-attach from scratch â€” never re-sync a stalled RSP, Â§5.4/Â§9.3), `debug.kdb`, and `debug.introspect.run`. Every other `GdbMiError` keeps the Phase-C contained-error behaviour (session stays live, agent may retry or `end_session`). The detect-and-report is bounded by the existing MI/interrupt timeouts, so the tool call returns within its ceiling and the target is provably resumed; it never hangs.

## Consequences

- An induced stall (a controller whose `write`/`read` times out mid-session) surfaces as a single `INFRASTRUCTURE_FAILURE` (`transport_stall`) response with the guest resumed, the durable record written back to EXECUTING, the `StopCapableGuard` released, and the session reaped â€” re-attach starts clean and `target.run_tests` is ungated, matching the "never re-sync RSP" contract.
- A benign command error (bad symbol, bad address) is unchanged: contained, session kept, retryable.
- `remotetimeout` and the connect retry make the clean QEMU path more tolerant of a slow first packet without changing its success behaviour; the retry seam keeps unit tests instant.
- The classification is a property of the engine's raised `GdbMiError.details["code"]`, so it is testable at the handler boundary with a fake controller that raises a stall vs a plain `^error`.

## Considered & rejected

- **Treat every `GdbMiError` timeout as a stall and always tear down.** Rejected: a transient command timeout on an otherwise-live session would needlessly destroy a recoverable session; only a stall on the *link* warrants teardown, and the `code="transport_stall"` marker draws that line.
- **Auto-reconnect/re-sync RSP after a stall and replay the breakpoint ledger.** Rejected by the interface contract (Â§5.4, Â§9.3): a stalled or reset link fully invalidates the session; re-syncing risks driving a half-dead stub. Re-attach from scratch is the only sound recovery, so the handler reaps and points the agent at `debug.start_session`.
- **Retry interactive verbs (continue/step) on timeout.** Rejected: `-exec-continue` already resumed the target before the stall; re-issuing it would double-resume or fight the interrupt fallback. Only the idempotent connect is retried.
- **Classify transient vs permanent connect failures by matching gdb's `^error` message text.** Rejected: the connect-error wording is gdb-build/locale-dependent, so substring matching is brittle and silently wrong when the text shifts. The connect is idempotent until `^connected`, so retrying *any* connect error a bounded number of times is correct without classification â€” a permanent failure just costs `count Ă— backoff` before reporting the same clean error.
- **Leave `remotetimeout` at gdb's default.** Rejected: the default is short and build-dependent; over a serial line it produces spurious drops or opaque stalls. An explicit generous-but-finite value makes the wait predictable and gdb-reported.
- **Add a new `TRANSPORT_STALL` `ErrorCategory`.** Rejected: `INFRASTRUCTURE_FAILURE` already denotes "the plumbing failed, not your input"; the `code="transport_stall"` detail gives the specific signal without growing the agent-facing taxonomy.
