# ADR 0019 — `debug.gdb` KGDB/RSP tier: a persistent gdb/MI engine (replacing the batch text-scraper), delivered as a phased sub-issue series

**Status:** Accepted (2026-05-29) · **Issue:** #13 (epic #9; depends on #10 transport [closed] + the #17 hardening set [#65/#66/#67/#68/#69/#70/#71, all merged]; consumes ADR 0017 symbol version-lock and ADR 0018 break policy) · **Affects:** a new `providers/gdb_mi.py` (persistent `gdb --interpreter=mi3` runner + `pygdbmi` parse layer) replacing the per-operation `-batch` paths in `providers/qemu_gdbstub.py`; the existing `debug.*` tool surface in `server.py` (`debug.start_session`/`set_breakpoint`/`continue`/`read_memory`/`read_registers`/`evaluate`/`step`/`backtrace`/…) re-pointed at the new engine; `pyproject.toml` (new `pygdbmi` runtime dependency); `prereqs/checks.py` (gdb MI-capability probe). No change to the transport, guard, or break-policy seams — those are owned by #10/#17.

## Context

Issue #13 adds the full source-level KGDB/RSP debug tier (`debug.gdb`): breakpoints, single-step, locals, backtraces, register/memory access over the gdb Remote Serial Protocol, for both the QEMU gdbstub (RSP over TCP) and serial KGDB (RSP demuxed off the console by `agent-proxy`/`kdmx`, #10) transports.

Two facts shape the work:

1. **A `debug.*` surface already exists** from the Phase-4 live-debug MVP (#5): `debug.start_session`, `debug.set_breakpoint`, `debug.continue`, `debug.read_memory`, `debug.read_registers`, `debug.evaluate`, `debug.interrupt`, `debug.end_session`, etc. (`server.py:6621–6781`). But it drives gdb in **one-shot batch mode** — `argv = [gdb_path, "-nx", "-batch", "-q"]` (`providers/qemu_gdbstub.py:305,786`) — launching a fresh gdb per operation and parsing **human-formatted** output. That is precisely the failure mode #13 exists to remove: an agent (or our parser) misreading free-form gdb text. It also cannot hold breakpoints/stepping state across calls, because each call is a new process.

2. **#13 is unblocked.** Its dependency #10 (transport abstraction + `rsp_endpoint`) is closed (`transport/{base,proxy,serial_local}.py`), and the entire #17 hardening set is merged: `StopCapableGuard` (#68, ADR 0015), `SessionGuard` (#66, ADR 0013), watchdog relax/restore (#69, ADR 0016), symbol version-lock (#70, ADR 0017), break policy (#71, ADR 0018), secrets/redaction (#65, ADR 0012), IPMI cipher policy (#67, ADR 0014). The seams `debug.gdb` must call already exist — `debug.start_session`'s `transport_enabled` branch already opens a guard-protected `TransportSession` and an integration test covers it (see the spec's "Prerequisites (verified)"). One residual is *not* external: the provider still attaches gdb to the **raw boot endpoint**, not `TransportSession.rsp_endpoint`, so re-pointing the engine at the transport session's RSP endpoint is in-scope Phase A work.

So #13 is not a greenfield add — it is an **engine replacement plus a migration** of an existing tool surface, gated by guards that already exist, across two transports, plus module-symbol loading and RSP-stall robustness. The sibling tier #11 (`debug.introspect`) was decomposed into #51–#56 "so each can ship as a focused PR"; #13 is larger and more transport-sensitive and warrants the same treatment.

## Decision

1. **Replace the batch engine with a persistent gdb/MI engine.** Introduce `providers/gdb_mi.py`: a long-lived `gdb --interpreter=mi3` subprocess whose MI records are parsed by **`pygdbmi`** into typed JSON. RSP attach is `-target-select remote <host>:<port>` against the transport's `rsp_endpoint` (§3.3, `required_caps=[provides_rsp]`). Per "replace, don't deprecate" (global standards), the `-batch` text-scraping paths in `qemu_gdbstub.py` are **removed** in the same series, not left as a parallel engine.

2. **Migrate the existing `debug.*` tools in place.** The current `debug.*` names *are* the gdb tier; they keep their names and are re-pointed at the MI engine, gaining the missing structured operations (`step`/`next`/`finish`, `backtrace`/frames, `list_variables`). No parallel `debug.gdb.*` namespace is introduced.

3. **Add `pygdbmi` as a runtime dependency.** MI3 is a structured-but-nontrivial grammar; `pygdbmi` is its canonical parser. Hand-rolling the parser is rejected (below). The dep is introduced in Phase A so its supply-chain review is isolated.

4. **Deliver as a four-phase sub-issue series under #13.** #13 stays open as the umbrella tracking issue (mirroring #11). Phase boundaries are chosen so each lands behind the constrained `ALLOWED_DEBUG_OPERATIONS` gate and is independently reviewable/revertible:

   - **Phase A — MI engine foundation.** Persistent `gdb -i=mi3` runner + `pygdbmi` parse layer; add the dep; RSP connect over **`TransportSession.rsp_endpoint`** — the residual gap is that the legacy provider still drives gdb against the *raw boot endpoint* (`server.py:4124,4233`; `qemu_gdbstub.py:288-302`), so Phase A re-points the engine at the guard-protected transport session; attach/detach lifecycle; **`StopCapableGuard` acquisition via `transport.open()`** (§5.3) on *every* transport including `qemu-gdbstub`; guaranteed best-effort `continue` + report on error (no target left `HALTED`). gdb MI-capability probe in `host.check_prerequisites` (today only gdb *presence* is checked, `prereqs/checks.py:48`): pin a **minimum gdb 9.1** (the GDB manual's documented `mi3` introduction), probe that **runs one mi3 MI command and gets a valid `^done` record** (not merely that the `mi3` name is accepted), and **hard-fail with a version-naming message** on older or `mi3`-less gdb — `mi3` is required, no `mi2` fallback.
   - **Phase B — symbols & provenance.** `vmlinux` symbol load; `KernelProvenance` `build_id` version-lock verified **pre-attach in the handler, not over RSP** (consumes ADR 0017 / #70); fail loud on mismatch. *(Amended 2026-05-29, ADR [0020](0020-gdb-mi-symbol-resolution-mechanism.md): Phase B also introduces an **internal** `-data-evaluate-expression "&<name>"` call to resolve one validated canonical symbol — this is the engine's first use of that verb, predating the Phase C migration below. It is internal and name-shape-gated, not an agent-facing or arbitrary-expression surface, so the constraint in the Phase C bullet and the rejected alternative is unchanged.)*
   - **Phase C — core operations, MI-typed.** Migrate breakpoints/watchpoints, `continue`, `step`/`next`/`finish`, `-stack-list-frames`, `-stack-list-variables`, register/memory reads onto MI typed JSON; **delete** the corresponding batch-mode paths. `-data-evaluate-expression` is used **only internally** to implement the existing named inspectors (`kernel_version`, `symbol_address`); the `debug.evaluate` allowlist is preserved exactly and no arbitrary-expression capability is exposed (see rejected alternative below).
   - **Phase D — module symbols, robustness, serial transport.** Per-module section-address discovery + `add-symbol-file` at runtime addresses; `set remotetimeout`, retry/backoff, transport-stall detect-and-report (never hang); break entry via `transport.inject_break` using ADR 0018's `BreakPolicy`; serial-KGDB (demuxed) smoke test + transport-quality warning surfacing (SOL/vterm → suggest `debug.kdb`/`debug.introspect`); `docs/debug-gdb.md`.

5. **Record the design in a spec.** `docs/archive/superpowers/specs/2026-05-29-debug-gdb-mi-tier-design.md` carries the full phase breakdown, the per-phase scope/acceptance/coordination, and the sibling-issue map; this ADR records only the decisions and their rejected alternatives.

## Consequences

- The agent-facing `debug.*` contract gains structured step/frame/variable operations and stops depending on free-form gdb text; results are MI-derived typed JSON.
- One engine, not two: the batch paths in `qemu_gdbstub.py` are deleted by end of Phase C, so there is a window (after A/B, before C completes) where the new engine and the old batch paths coexist on a branch — acceptable because the series merges in order and the gate keeps unfinished operations unreachable.
- A new runtime dependency (`pygdbmi`) enters the tree, which until now carried only `mcp` + `pydantic`; it gets pinned and supply-chain-reviewed in Phase A.
- `debug.gdb` and `debug.kdb` (#12) remain mutually exclusive per target via the existing `StopCapableGuard`; this series does not change that invariant, only adds a second acquirer.
- A provisioning `reset()` (incl. `mode=kexec`) fully invalidates an open session (§5.4, §9.3): the engine must re-attach from scratch, never re-sync RSP.

## Considered & rejected

- **One big-bang PR for all of #13.** Rejected: it bundles a new runtime dependency, an engine rewrite, an in-place migration of ~10 shipped tools, two transports, and integration tests into a single review surface — against the #11→#51–#56 precedent and the repo's adversarial-review-per-decided-artifact workflow. Hard to review, hard to revert.
- **Keep the batch engine; add the MI engine alongside it.** Rejected: violates "replace, don't deprecate." Two engines doubles maintenance, splits behavior, and invites drift between the text-scraped and MI-parsed results for the same operation.
- **Hand-roll the MI parser (no new dependency).** Rejected: MI3's grammar (async `*stopped`/`=` records, tuple/list values, token correlation) is enough surface to get subtly wrong, and a bespoke parser is exactly the "misread gdb output" risk #13 targets. `pygdbmi` is the canonical, widely-used parser; the dependency cost is the smaller risk and is reviewed in isolation in Phase A.
- **A new `debug.gdb.*` tool namespace parallel to the existing `debug.*`.** Rejected: the existing `debug.*` surface already *is* the gdb/RSP tier; a parallel namespace would create two ways to do the same thing, force a deprecation path, and churn every caller. Migrate in place instead.
- **Drive gdb via the Python `gdb` module / scripted CLI instead of MI.** Rejected: the in-process `gdb` Python API isn't available when driving a remote stub from a managed subprocess, and scripted-CLI output is the same free-form text we are trying to stop parsing. MI is the structured machine interface for exactly this use.
- **Verify symbol/`build_id` provenance over the live RSP connection.** Rejected here as it was in ADR 0017: provenance is verified pre-attach in the handler against the artifact store; doing it over RSP is slower, racier, and couples version-lock to a live transport.
- **Expose arbitrary `-data-evaluate-expression` as an agent-facing op.** Rejected: it would regress the constrained-debug-surface invariant (CLAUDE.md, `ALLOWED_DEBUG_OPERATIONS`). The existing `debug.evaluate` accepts only the named inspectors `kernel_version`/`symbol_address` and rejects everything else (`qemu_gdbstub.py:497-546`); MI eval stays behind that inspector allowlist as an internal implementation detail, not a raw-expression escape hatch.
