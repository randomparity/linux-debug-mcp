## Risk surfaces inspected even where no test failed

The spec's "Known precondition" section flagged three risk surfaces to
inspect proactively even if no test failure pointed at them. Reviewing
each:

- **`/proc/net/tcp` and `/proc/net/tcp6` parsing in
  `seams/process_identity.py`**: the runner produces parseable
  `/proc/net/tcp` and the test that actually round-trips a listener
  (the `os.getpid()` arm of row 1) passes — so the parser is correct
  on the runner. The failure in row 1 is in the *fd-inspection* branch
  downstream of the parser, not the parser itself. No parser changes
  needed.

- **PTY allocation and `flock` behavior under
  `transport/serial_local`**: the PTY/flock-touching tests
  (`test_serial_local_transport_integration.py`,
  `test_transport_open_close_integration.py`) are skipped on the
  public runner because they require `agent-proxy` — the kernel-level
  PTY/flock behavior is exercised only on the self-hosted runner via
  `transport-integration.yml`. No drift observed in the unit tests
  that DO run on the public runner (`test_serial_local_transport.py`,
  `test_break_inject.py`, `test_transport_proxy.py` etc.).

- **`/proc/<pid>/*` walkers (cgroup hierarchy, `/proc/<pid>/fd`,
  `/proc/<pid>/status` layout)**: the runner is cgroup-v2 (Ubuntu
  24.04 default); none of the unit tests assume v1. The
  `/proc/<pid>/fd` permission boundary is *exactly* what row 1 trips
  over — that finding is captured in row 1's disposition, not as a
  separate risk-surface entry.
